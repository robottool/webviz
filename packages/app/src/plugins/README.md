# 3D display plugins

Deep reference for the 3D-tab display plugins and their supporting `core/` helpers.
`CLAUDE.md` (repo root) carries a one-line summary of each and points here for detail.

The `DisplayPlugin` contract + `PluginContext` (`{ hub, tf, scene }`) + `pluginRegistry` live in
`core/plugin.ts`; `plugins/index.ts` registers built-ins (imported for side effect before the
catalogue is read). Every consumer plugin anchors its geometry to a payload `frame_id` via the
shared `TFManager`'s `resolveToFixed` (the scene root *is* the fixed frame); unresolved frames hide.

## RobotModelPlugin

`plugins/RobotModelPlugin.ts` — renders a URDF via `urdf-loader` with three independently switchable input sources: **URDF** (`local` folder upload **or** a `wv/RobotModel` channel), **joints** (from a `wv/JointState` channel — an articulated robot stays **hidden until its first live joint state arrives** rather than showing a misleading all-zero pose (`isAwaitingJoints()` drives a Properties "waiting for live joint states" hint; the first sample `emitChange`s to clear it); a jointless/static robot shows immediately; interactive joint control lives in jog mode), **pose** (base placed via TF/`resolveToFixed` on a base frame picked from the live TF frame list — identity until it resolves). Both the joints channel and base-frame picker sit under one collapsible **"Live state"** Properties section (URDF + Live state auto-collapse once the robot is cleanly loaded); defaults to all-channel so a published robot auto-displays. (The manual joint/pose inputs were removed — interactive control lives in jog mode.) Emits a validation report (joints + limits, meshes loaded/failed) consumed by `RobotModelProperties`.

Four load entry points, all converging on the `reloadLocal` pipeline where local: channel-published meshes load cross-origin when the `wv/RobotModel`'s `urdf_url` is an absolute http(s) URL (`loadFromChannel` sets `remoteUrdfBase = repoBaseOf(url)`, resolving `package://` mesh refs against the repo base — same as a URL load), and from the hub asset server (anchored on `ur_description/`) for a relative/hub-served `urdf_url`; local meshes resolve to blob URLs. `loadFromManifest(baseUrl, paths)` **fetches** a file set over relative HTTP (used by the Properties "load demo robot" button, which loads the bundled primitive-geometry `packages/app/public/demo-robot/demo_arm.urdf`) — it converges on the same `reloadLocal` pipeline as the picker, so `localFiles` is `Array<File | PickedFile>` and access goes through `pathOfLocal`/`textOfLocal`. `loadFromUrdfUrl(input)` loads a URDF from a **GitHub/raw URL** (`toRawUrl` normalizes `github.com/blob` → `raw.githubusercontent.com`; `repoBaseOf` derives the repo base; `resolveRepoMeshUrl` maps the URDF's `package://<pkg>/<rest>` mesh refs to `<base><rest>`, the common `*_description` layout) — `urdf-loader` then fetches each mesh on demand cross-origin, so it only works for CORS-enabled hosts and flat (non-xacro) URDFs. Mesh resolution in `loadMesh` is three-way: local blob resolver, else `remoteUrdfBase` (URL load **or** a channel load whose `urdf_url` is absolute http(s)), else the hub asset server (relative/hub-served channel load). This whole local path is fully client-side (no hub), so it's the one feature that works on a hub-less static deploy. The Properties "Load URDF…" button opens a dialog with all three sources (folder · URL · demo) and surfaces load errors.

### Jog mode + IK

Beyond the monitor joints (channel/manual) there's a separate **`jog` toggle** (the base `robot` stays the live-state **monitor**; jog spawns a translucent **shadow** clone as the interactive command target). Enabling jog `clone(true)`s the robot (with independent geometry + faded materials so the ghost renders/disposes without touching the monitor), adds it as `${id}:jog`, **freezes its base**, and drops a move+rotate gizmo on a chosen TCP link — so you can drive the shadow while the solid robot keeps showing current state (`onRender` always drives the monitor; the controller owns the shadow); dragging it runs a damped-least-squares Jacobian solve (`plugins/RobotIkController.ts` + `core/ik.ts`) and re-poses the arm to follow. The controller (`plugins/RobotIkController.ts`) builds the actuated chain root→TCP by walking `urdf-loader`'s scene-graph parents, and owns the gizmo; while jogging, the controller owns the shadow's joints and its base stays frozen (the monitor robot is driven independently by `onRender`).

It has **two interchangeable solver backends** (`ik_backend`): **`native`** solves in-browser — FK from each joint's `matrixWorld`, an analytic geometric Jacobian from its world axis/origin, the DLS step from `core/ik.ts`, applied via `robot.setJointValue` (fully client-side, no hub) — the drag is a **preview** that publishes nothing, plus an on-demand **"Send to robot"** button (`RobotIkController.sendTarget()`) that publishes *once* both the current TCP pose (`wv/Pose` on `ik_target_channel`) and the previewed joint config (`wv/JointState` on `ik_solution_channel`) and **holds** both (a keepalive re-asserts those snapshots, so dragging afterwards doesn't move the robot until the next send); **`external`** publishes the gizmo pose as a `wv/Pose` target (`ik_target_channel`, default `tcp_target`) via `core/sourcePublisher.ts` (rate-capped drag + a `setInterval` keepalive so a late solver latches) and drives the robot from a `wv/JointState` **solution** channel (`ik_solution_channel`, default `ik/joint_states`) the user's own solver (MoveIt/KDL/ikfast/…) publishes back — so the drag-TCP UX stays but the kinematics are theirs (needs a hub). Either backend reports a reach residual (FK-vs-target); external shows "waiting for solver" until the first solution.

The jog toggle is only offered for **arm-like** robots: `isIkFeasible()` (cached at load) requires a serial chain with ≥ `MIN_IK_DOF` (2) actuated joints, so a mobile base / static / single-joint model hides it (and jog auto-disables when loading an infeasible robot). A legacy persisted `joint_source: 'ik'` migrates to `joint_source: 'channel'` + `jog: true` on construction. Properties (`RobotModelProperties.tsx`'s `IkPanel`) exposes the TCP-link picker, the solver toggle (native: orientation-weight slider · external: target/solution channel inputs), and the reach/residual readout. (`RobotIkController.reseed()` still re-homes the gizmo to the current TCP — used on entering jog and after a joint nudge — but there's no longer a manual "Recenter" button; toggling jog off/on rebuilds the shadow from the monitor.) The per-joint limit sliders (the shared `JointSliders` component, also used by Manual mode) stay available in IK mode as a "fine-tune" section — they live-track the solved values (`plugin.getJointValue`) and nudging one calls `plugin.setIkJoint` → sets the joint + `reseed()`s the gizmo to the new TCP. A joint the solver has pinned at a limit *while the target can't be reached* is flagged with a ⚠ on its slider (`RobotIkController.computeAtLimit` → `getJointsAtLimit` → `plugin.getIkJointsAtLimit`), so you can see why the arm stopped following the gizmo. Below the joints is a **TCP nudge** panel (`TcpNudge`) — xyz(m)+rpy(°) rows, each an input + −/+ step buttons — that reads/writes the gizmo target pose via `plugin.getIkTcpPose`/`setIkTcpPose` (→ `RobotIkController.getTcpPose`/`setTcpPose`, which re-solve). Both the Joints and TCP-nudge sub-panels are collapsible, as are the URDF + Live-state sections (`sectionToggle`/`collapsed` map; URDF+Live auto-collapse on a clean load).

### Demo mode

**Demo mode** (global `settings.demoMode`): the plugin delegates the whole lifecycle to `plugins/RobotDemoController.ts` (a sibling of `RobotIkController`, reaching back into the plugin only through a small `DemoHost` adapter — no import cycle), which subscribes to the flag and, when on and a robot is loaded, spins up a `core/demoExecutor.ts` **DemoExecutor** node — a client-side stand-in for a robot controller. It `advertiseLocal`s the dummy's *feedback* channels `demo/joint_states` (`wv/JointState`) + `demo/base_frame` (`wv/Transform`, identity placing the base link at the fixed frame), points the monitor's `joint_channel`/`root_frame` at them, and **subscribes to the command channel** (`ik_solution_channel`) that jog's "Send to robot" publishes — animating `demo/joint_states` toward each new goal over ~0.8s (deduping the held/keepalive re-asserts), exactly like a real controller executing a commanded motion.

**URDF geometry source (local files / URL / channel) is orthogonal to demo mode** — local loading and jog/drag-IK are *not* gated; only the fake live-state is. So the real-robot IK workflow works with a locally-loaded URDF too: load local URDF (or a `wv/RobotModel` channel) → live joints from a `wv/JointState` channel → jog → command via the native one-shot "Send to robot" publish or the external solver round-trip (needs a hub). The channel URDF path (`loadFromChannel`) currently uses only `urdf_url` (URDF+meshes fetched from the hub asset server or a CORS URL); inline `urdf_xml` isn't wired up. `sendIkTarget` is **uniform** — always `RobotIkController.sendTarget()`, no demo branch — so demo and real robots receive the *same topics*; only the **transport** differs, chosen once at `enterJog` via `ikPublisher()`: an in-app `HubClient.advertiseLocal` channel in demo mode (the DemoExecutor consumes it, no hub) vs the hub source socket (`core/sourcePublisher.ts`) otherwise, where a real robot receives the same command. `RobotIkController` takes that as an injected `Publisher` factory rather than importing `sourcePublisher` directly. Because the demo path is all local channels, it works on the hub-less static deploy. Toggling it from the ⚙ panel (`TopBar` `toggleDemo`) resets the workspace **either way** (`newWorkspace` — closes every panel) so nothing conflicts with, or lingers after, the demo; it confirms first when there's work to lose (open panels, plus live hub data when entering). Turning demo mode **off tears the whole scenario down** (`stopDemo`): stops the fake publisher, disables jog, unloads the URDF, and clears the live-state binding (`clearRobot`) — then closes every panel (the off path flips the flag *before* `newWorkspace` so `stopDemo` runs while the RobotModel panel is still mounted).

## Consumer plugins

- `plugins/TFFramesPlugin.ts` — visualizes the TF tree: an `AxesHelper` triad + camera-facing canvas-texture label per frame, plus parent→child line segments. A **pure TFManager consumer** — no channel binding of its own; each frame is placed via `resolveToFixed` (the scene root *is* the fixed frame), unresolved frames are hidden. Uses the generic `PropSchema` Properties form (source filter, frame-name filter, axis scale, labels, parent links); a link is drawn only when both endpoints pass the filter.
- `plugins/MarkerPlugin.ts` — subscribes to one `wv/Marker` channel and keeps a marker **store** keyed by `namespace/id`, honoring the action lifecycle (`add`/`modify`/`delete`/`delete_namespace`/`delete_all`) and `lifetime` expiry. Each marker is a group anchored to its `frame_id` via `resolveToFixed` each render, with the geometry offset inside by the marker's own pose. Geometry: cube, sphere, cylinder, arrow, line_strip, line_list, points, triangle_list, and `mesh` (loaded asynchronously from the hub asset server via `core/meshLoad.ts`). The empty group/entry is created synchronously so the lifecycle tracks the identity, and the loaded object is attached only if the entry is still the live one for its key — guarding the replace/delete-mid-load race. A re-published mesh marker with the **same** `mesh_url` updates in place (pose/scale/color) rather than refetching, so a high-rate (e.g. 20 Hz) stream doesn't reload the resource every frame; `scale` multiplies the mesh's intrinsic `baseScale` (captured at load, so updates don't compound) and color is only re-tinted when it changes (`colorKey`). A non-zero marker `color` tints the whole mesh, an all-zero color keeps the mesh's embedded materials. `text` (needs a CSS2D renderer) is still deferred and ignored.
- `plugins/PointCloudPlugin.ts` + `plugins/pointcloud.worker.ts` + `core/pointcloudDecode.ts` — the performance-critical plugin (§10). Binary `wv/PointCloud` frames are **decoded in a WebWorker** (`pointcloud.worker.ts`, instantiated via Vite's `new Worker(new URL(...), { type: 'module' })`); the pure deinterleave lives in `pointcloudDecode.ts` (no three/Worker globals, so it's node-testable and shared by both). The plugin copies each payload out of the shared frame buffer, transfers it to the worker, and on the result swaps the buffers into a BufferGeometry. Coloring: RGB → vertex colors; intensity → viridis colormap (`core/colormap.ts` DataTexture, sampled in a `ShaderMaterial`) normalized per frame; xyz-only → colormap over z. Anchored to the payload's `frame_id` via `resolveToFixed`.
- `plugins/{LaserScan,OccupancyGrid,Path,Pose}Plugin.ts` — the JSON sensor/nav plugins, each subscribing to one channel of its schema and anchoring to `frame_id` via `resolveToFixed`. **LaserScan**: polar→Cartesian `THREE.Points` (drops out-of-range / `"Inf"` beams). **OccupancyGrid**: base64 uint8 → `DataTexture` (free=white, occupied=black, unknown=transparent) on a `PlaneGeometry` sized `w·h·resolution`, nested root(frame)▸origin(grid origin)▸plane. **Path**: `THREE.Line` polyline colored by the path's RGBA. **Pose**: an `ArrowHelper` along +X plus an optional 2σ covariance ellipse (`ellipse2D` does the 2×2 eigen of the position block).

## Interactive producer

- `plugins/CoordinateFramePlugin.ts` — the one **interactive** display (a *producer*, not a consumer). Shows a combined move+rotate gizmo the user manipulates to author a 6-DOF pose — e.g. an IK / TCP target — and **publishes** it as a `wv/Pose` channel (default name `tcp_target`) back to the hub via `core/sourcePublisher.ts`. three.js `TransformControls` (same `three/examples/jsm/controls` dir as `OrbitControls`) renders one mode at a time, so to show translate axes *and* rotate rings together it attaches **two** TransformControls to the same `gizmoNode` — one `translate`, one `rotate`; there is no separate triad mesh, the gizmo handles *are* the visual. The **rotate control is constructed first on purpose**: each TransformControls grabs pointer capture in listener (construction) order and the `dragging-changed` handler disables the *other* control while one drags, so the first-built wins any handle overlap — rotation gets priority (the move arrows are still grabbable on their outer, ring-free segments). `pruneGizmo` strips handles from each control's internal `_gizmo` (both the visible gizmo and the raycast picker): translate loses the `XY`/`YZ`/`XZ` plane rectangles — **removing their large pickers is what lets clicks reach the rotate rings** (otherwise the plane pickers swallow them and rotation appears dead) — and rotate loses the big outer `XYZE`/`E` rings, leaving just the X/Y/Z axis rings. The node lives under the scene root (= the fixed frame, at world identity), so its transform is the fixed-frame pose, published as the Pose's `frame_id` (`scene.getFixedFrame()`). On `dragging-changed` it freezes `OrbitControls` **and the other gizmo** so they don't fight; `objectChange` syncs the pose into settings (so the generic Properties form's X/Y/Z + roll/pitch/yaw fields reflect drags on its next render — and typing those fields drives the gizmo back). `onRender` publishes rate-capped while dragging (`PUBLISH_HZ`) with a low-rate keepalive (`KEEPALIVE_MS`) so late subscribers latch the current pose. the gizmo is always **local space** (`tc.setSpace('local')`) so the handles follow the frame's orientation — rotating a ring visibly turns the move arrows with it; gizmo `size` is a setting and the rotate rings are drawn at `RING_SCALE×` the move-arrow size.

## Supporting `core/` helpers

- `core/sourcePublisher.ts` — lets the app publish data *back* to the hub as a second source (`role=source&id=ui`), the same coexists-with-a-live-connection trick as `core/player.ts`. The app's `HubClient` is a consumer (`role=client`) and can't advertise, so this singleton owns one shared source socket per window; `advertise(name, schema)` returns a `PublishHandle` with `send`/`close`. The hub remaps `(conn, localId)` → fresh global id and rewrites the frame `channel_id` (`channel_registry.ts`), so subscribers — including our own `HubClient` — see a new named channel. Each channel's last value is **latched** and re-sent on (re)connect. Currently the lone producer is `CoordinateFramePlugin`.
- `core/demoExecutor.ts` — the client-side **fake robot controller** for RobotModel demo mode. Advertises the dummy's feedback channels `demo/joint_states` + `demo/base_frame` via `HubClient.advertiseLocal` (no hub, no socket) and **subscribes to the command channel** ("Send to robot"'s `wv/JointState` goal); on each new goal it interpolates `demo/joint_states` from the current to the commanded config over ~0.8s, so the monitor robot animates as if a real controller executed the motion. Deduplicates the held/keepalive command re-asserts. Driven by `RobotDemoController` (created/torn down with the global `demoMode` toggle).
- `core/meshResolver.ts` — `LocalAssetResolver`: indexes folder-picked files by `webkitRelativePath`, resolves `package://`/relative refs by **longest trailing-segment match**, and exposes a `THREE.LoadingManager` whose `setURLModifier` routes every loader request (mesh + DAE textures / GLTF `.bin`) to a blob URL.
- `core/meshLoad.ts` — shared mesh loading for `RobotModelPlugin` and `MarkerPlugin`'s `mesh` type: `loadMeshFromUrl(url, format, material?, manager?)` dispatches to the right three.js loader by extension (`.dae`/`.stl`/`.glb`/`.gltf`), and `resolveAssetUrl(path)` anchors `package://`/relative refs onto the hub asset server (`http://<host>:8080/assets`). RobotModel keeps its own URL resolver for the UR `package://example-robot-data/...ur_description/` anchoring quirk; the generic `resolveAssetUrl` just strips the scheme and prepends the asset base.
- `core/ik.ts` — pure damped-least-squares (Levenberg–Marquardt) IK math for RobotModel's jog/IK: `orientationError` (quaternion → axis·angle rotation vector), `solveDampedLeastSquares(J 6×N, e 6, λ)` → joint delta `dq = Jᵀ(J Jᵀ + λ²I)⁻¹e` via a small `solveLinear` (Gauss–Jordan). Only three.js vectors + number arrays, so it's node-testable and framework-light.
- `core/poseGizmo.ts` — reusable combined move+rotate gizmo (two `TransformControls`, rotate built first for pick priority, plane/outer-ring pickers pruned, `local` space), extracted from the `CoordinateFramePlugin` pattern. Used by `RobotIkController`; `CoordinateFramePlugin` still has its own inline copy (could adopt this later).
- `core/uuid.ts` — context-safe UUID v4 used to mint tab/plot ids (`tabs.store`, `PlotTab`, `ThreeDTab`). Falls back from `crypto.randomUUID()` (secure-context only) to `getRandomValues` to `Math.random`, so it works when the app is served over plain HTTP from a LAN/VM address.
