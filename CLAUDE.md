## What this is

WebViz is a browser-based visualization platform for robots and real-time systems. Three principles drive the design (`webviz_design_doc_v3.md` is the spec; code comments cite its sections as `┬зN`):

- **Protocol-first** тАФ the `wv/*` wire protocol (`packages/protocol`) is the single contract shared by hub, app, and SDKs. Change it there and only there.
- **Source-agnostic** тАФ the browser reasons only about channel *names* + schema types, never about where data came from (ROS, file, custom SDK). The hub erases origin.
- **Tiling-panel workspace** тАФ the app is a recursive split layout of independent panels sharing one connection (no tab bar; split to add a panel, close to remove, maximize to focus). "Switch between whole setups" is named layouts, not tabs.

The repo is a **vertical slice in progress**. **All six tabs are live** тАФ **Inspector**, **3D**, **Image**, **Plot**, **Map**, and **Log** тАФ and the 3D display-plugin catalogue is essentially complete: `RobotModel`, `TFFrames`, `Marker`, `PointCloud`, `LaserScan`, `OccupancyGrid`, `Path`, `Pose`, `CoordinateFrame` (plus TFManager and SceneManager). Recording **capture *and* playback** both work now (`.wvrec` capture + replay-as-a-source). The **Python SDK**, a **ROS 2 adapter** (`sdks/ros2`), and a header-only **C++ SDK** (`sdks/cpp`, ┬з6.3) all exist тАФ the design doc's SDK catalogue is complete.

**Static (hub-less) demo build.** `WEBVIZ_BASE=/<repo>/ pnpm --filter @webviz/app build` emits a static bundle whose asset URLs are rooted at `/<repo>/` for GitHub Pages (`.github/workflows/pages.yml` deploys it on push to `main`). `base` defaults to `/` (env-gated in `vite.config.ts`), so the hub-served build and dev server are unchanged. ## Conventions

Non-obvious rules that hold repo-wide тАФ worth knowing before editing:

- **`.js` import extensions on `.ts` sources.** Intra-package imports write `./foo.js` even though the file is `foo.ts` (ESM + `moduleResolution: Bundler`). Match the surrounding code.
- **The binary payload layout is duplicated four times, by design.** `protocol/src/binary.ts` is the source of truth; `sdks/python`'s `client.py` (+ the demos), the ROS 2 adapter's `converters.py`, and the C++ SDK's `client.hpp` each re-encode the same bytes. Change one тЖТ change all four (see [When changing the protocol](#when-changing-the-protocol)); the C++ `test/test_payloads.cpp` and protocol vitest guard the contract.
- **Little-endian is a WebViz decision** (the wire spec doesn't mandate one) тАФ keep every encoder/decoder in sync.
- **Ordering keys on the source clock, never browser wall-clock** (`TimeManager` `maxSeenT`) тАФ skew-safe in a VM.
- **Stores avoid import cycles with one-way deps** тАФ e.g. `settings.store` is *subscribed to* (never imports `connection.store`); `RobotDemoController` reaches the plugin only through a small adapter.
- **Pure/framework-light modules are kept node-testable** (no three/Worker/rclpy globals): `core/pointcloudDecode.ts`, `core/ik.ts`, `core/plotSeries.ts`, the ROS `converters.py`. Keep new pure logic that way so it can be unit-tested.
- **Design-doc citations.** Code comments cite `webviz_design_doc_v3.md` sections as `┬зN`.

## Architecture

Three TS packages + a Python SDK, pnpm workspace (`packages/*`), ESM throughout (`"type": "module"`), strict TS.

### `packages/protocol` тАФ the wire contract
- `schemas.ts` тАФ `wv/*` schema types (Transform, Marker, Image, PointCloud, тАж): the shape of each channel's `data` payload.
- `messages.ts` тАФ JSON control/data ops over the WS text channel (`server_info`, `advertise`, `subscribe`, `message`, тАж), with discriminated-union types (`ClientMessage`, `SourceMessage`, `ServerMessage`, `AnyMessage`) for exhaustive routing.
- `binary.ts` тАФ binary data frames. **Fixed 20-byte little-endian header**: `op(0x01)` / 3 reserved / `uint32 channel_id` / `float64 timestamp` / `uint32 payload_length` / payload. Little-endian is a WebViz decision (the spec doesn't mandate one) тАФ keep encoder and decoder in sync. Also holds the schema-specific binary payload codecs: `wv/Image` and `wv/PointCloud` both lead with a length-prefixed `frame_id` (so the cloud/image carries its own TF frame); the PointCloud float region is generally unaligned, so the decoder copies it into an aligned buffer before viewing it as `Float32Array`.
- `frame.ts` тАФ `decodeFrame(raw)` normalizes any incoming WS frame (string JSON or binary) into a `data` or `control` result. The one decode entry point used by both hub and app.

### `packages/hub` тАФ broker + asset server (`server.ts`'s `startHub()` wires both, sharing one registry)
- `server.ts` тАФ `startHub(opts)`: the programmatic entry point. Constructs the `Broker` + asset server and resolves a `RunningHub` (`{wsPort, httpPort, close()}`) once HTTP is listening (rejects if a port can't bind). Keeping the wiring here тАФ not at module top-level тАФ lets a host embed the hub in-process. `main.ts` is the CLI shell that calls `startHub` with env-overridable ports/paths (`WEBVIZ_*`) and logs; the hub `package.json` `exports` `.` тЖТ `dist/server.js` so importers get `startHub` while `bin` still points at `main.js`. `Broker.close()` tears down the WS server for clean embedded shutdown.
- `broker.ts` тАФ WebSocketServer on :7777. Same port serves both roles, distinguished by query string `?role=source|client` (`&id=<sourceId>`). Sources advertise channels and push frames; the broker fans each frame out to subscribed clients, enforcing per-`(channel, client)` `max_hz` throttling тАФ **trailing-edge**: a frame arriving inside the throttle interval is parked (latest-wins) and flushed when the interval expires, so a throttled channel settles on its final value. **Backpressure**: data frames to a client whose socket has > 4 MB queued (`bufferedAmount`) are dropped, latest-wins, instead of buffering without bound (control messages are never dropped). **Latched channels** (`latched: true` in `advertise`, carried on `ChannelInfo`): the broker caches the latest frame per latched channel and replays it on `subscribe`, so late-joining clients see one-shot data (static maps, `/tf_static`, robot models); the cache is cleared on unadvertise/source disconnect. `injectJson()` lets a non-WS HTTP caller publish (used by `/api/inject`; pass `latched: true` in the body to latch).
- `channel_registry.ts` тАФ maps each source's *local* channel ids to *global* ids, renaming on collision (`resolveLocal`, `advertise`). This is the source-agnostic erasure layer.
- `session_store.ts` тАФ `SessionStore`: persists workspace layouts as JSON files under `dataDir` (`WEBVIZ_DATA_DIR`, default `data/layouts`), backing the `/api/layouts` REST endpoints. The hub is the shared/multi-user source of truth; the app also keeps a localStorage copy. Layout names are sanitized to a flat, safe filename namespace.
- `asset_server.ts` тАФ HTTP :8080: serves the built app, plus REST `GET /api/channels`, `GET/POST /api/layouts` (+ `/:id`), `POST /api/inject`, and `/assets/*`. Has path-traversal guarding for static serving. The `/assets/` prefix is shared by two roots тАФ robot descriptions/meshes (`assetsDir`) and the app's own Vite bundle (which Vite also emits under `/assets/`, i.e. `webDir/assets/*`) тАФ so it resolves `assetsDir` first, then falls back to the app bundle, then 404s.
- Ports/dirs are overridable via env: `WEBVIZ_WS_PORT`, `WEBVIZ_HTTP_PORT`, `WEBVIZ_WEB_DIR`, `WEBVIZ_ASSETS_DIR`, `WEBVIZ_DATA_DIR`, `ALLOWED_ORIGINS`.

### `packages/app` тАФ Vite + React + Zustand + three.js
The data path is **HubClient тЖТ TimeManager тЖТ MessageRouter тЖТ tab handlers**:
- `core/TimeManager.ts` тАФ the ┬з8 sync-window reorder buffer between the socket and the router. `HubClient` enqueues each decoded data frame; the TimeManager holds it for a small window (default 20 ms) and flushes in **source-timestamp order**, so a `wv/Transform` published alongside a sensor frame is dispatched (and TF-indexed) first. Ordering keys on the source clock (`maxSeenT`), never browser wall-clock (skew-safe in a VM); a wall-clock `maxHold` only force-flushes frames whose channel went quiet. `getCurrentTime()` exposes the latest source time.
- `core/recorder.ts` тАФ session recorder (┬з16.1). Taps the **raw** WS frames at the top of `HubClient.onMessage` (before decode), so a recording is byte-faithful, including binary `wv/PointCloud`/`wv/Image` frames. Records only what the browser actually receives (i.e. subscribed channels). `start(channels)` snapshots the live channel list as a **catalogue** (data frames carry only a numeric `channel_id`; the `advertise`/`server_info` that *name* a channel arrive at connect, before recording тАФ without the snapshot a mid-session recording couldn't be replayed under meaningful names). `stop()` serializes a `.wvrec` container (`WVR2` magic, then `uint32 catalogueLen` + JSON `ChannelInfo[]`, then per record `float64 t` / `uint8 type` / `uint32 len` / payload) as a Blob; the TopBar тП║ button downloads it. The reader still accepts legacy `WVR1` (no catalogue). Recordings are held in memory, so `capture()` enforces a **cap** (`MAX_BYTES` 256 MB / `MAX_FRAMES` 200k, whichever first): on hit it freezes the buffer and sets `isLimitReached()`, and the `RecordButton` poll auto-finalizes the download. The cap bounds memory and the cost of a backward seek in playback (which replays from the start тАФ kept lossless rather than per-channel-last-value, because accumulating channels like Markers' add/delete lifecycle and the Log stream would break).
- `core/player.ts` тАФ recording **playback** (┬з16.1), singleton `player`. Replays a `.wvrec` **as a second hub source** (`ws://<host>:7777?role=source&id=replay`) so it **coexists with a live connection** rather than replacing it. The trick: advertise each recorded channel using its *original recorded global id* as the source's **local** id, so the recorded frames replay **byte-for-byte unchanged** тАФ the hub remaps `(connId, localId) тЖТ fresh global id` and rewrites JSON `channel_id` / binary headers on the way out (`channel_registry.ts`). Replayed channels appear to the live `HubClient` as `replay/<name>`. Parses both `WVR2` (catalogue from the header) and `WVR1` (catalogue rebuilt from in-stream `advertise` frames); only `op:'message'` + binary data frames are replayed (control frames skipped). A rAF clock advances by `real╬Ф ├Ч speed` and flushes due records in timestamp order; transport API is `play/pause/seek/setSpeed/unload` (backward seek replays from t=0 so latest-value channels rebuild; `unload` closes the source socket тЖТ hub unadvertises the `replay/*` channels). `store/playback.store.ts` mirrors its state into React (like `connection.store`); `ui/PlaybackBar.tsx` is the bottom transport bar (тЦ╢/тП╕, scrubber, speed, тЬХ), rendered in `App.tsx` only when loaded; the TopBar ЁЯУВ button loads a file.
- `protocol/HubClient.ts` тАФ **singleton** (`hubClient`), one WebSocket per browser window. Owns reconnection, the live channel list (from `server_info`/`advertise`/`unadvertise`), and **reference-counted subscriptions by channel name**: first subscriber sends `subscribe`, last sender of unsubscribe sends `unsubscribe`. Subscribing to a not-yet-advertised channel is deferred until it appears. Subscribe by *name*, not id. Mirrors the last message per **latched** channel and replays it (async) to subscribers that bind *after* the hub's own replay already landed тАФ the hub only replays on the `subscribe` op, which ref-counting sends once per name. `advertiseLocal(name, schema)` injects a **client-side channel** (negative id, survives a `server_info` wipe) whose `send()` dispatches straight through the router тАФ indistinguishable from a hub data frame to subscribers/pickers/TFManager, but never hits the socket (used by demo mode; works with no hub).
- `protocol/MessageRouter.ts` тАФ fans one decoded frame out to every registered per-channel-id handler, so multiple tabs on the same channel each get their own callback.
- `store/connection.store.ts`, `store/tabs.store.ts` тАФ Zustand stores for connection state and the **panel workspace**. `TAB_META` defines the six panel types. `tabs.store` **persists the workspace to localStorage** (debounced) and hydrates on load. `connection.store`'s initial URL is `persistedHubUrl() || <auto-derived>` (settings override wins). The workspace is a **tiling layout** (`SplitState = {root, focusedLeaf, maximized}`), a **recursive binary tree** (`SplitNode` = `leaf{id,tabId}` | `split{id,dir:'row'|'col',frac,a,b}`). Each leaf pane *owns* one panel; `tabs` is the panel pool (a `TabConfig` per leaf), keyed by the leaf's `tabId` (null = empty pane awaiting a pick). Splitting replaces a leaf with `[old leaf, new **empty** leaf]`; closing collapses the parent to the sibling (closing the only pane clears it to one empty pane). Pure tree helpers (`splitLeafNode`/`removeLeafNode`/`mapLeaves`/`setFracNode`/`leafIds`/`findLeaf`) back the actions `splitPane(leafId,dir)`, `closePane(leafId)`, `setPaneType(leafId,type)` (the "pick content" action тАФ mints a panel, assigns it), `setSplitFrac(branchId,frac)`, `focusPane`, `toggleMaximize(leafId)`, plus `updateSettings`/`newWorkspace`. `gcPanels` (exported) drops panels no pane owns any more so the pool can't leak orphans; it runs after every structural change and on load. `sanitizeSplit(raw, tabs)` (exported) validates a persisted/loaded tree (drops stale tab idsтЖТnull, clamps fracs, regenerates missing node ids, validates `maximized`); no usable tree тЖТ one pane bound to the first panel (or empty). There is **no tab bar** (the old `ui/TabBar.tsx` and `addTab`/`activateTab`/`activeTabId` were removed in the panel pivot).
- `store/settings.store.ts` тАФ the few **global** knobs the тЪЩ panel exposes (most config is per-tab/per-display, not here): `theme` (`'telemetry'`|`'minimal'`|`'vibrant'`|`'industry'`, default `industry` тАФ the light theme) тЖТ `document.documentElement.dataset.theme`, `syncWindowMs` тЖТ `TimeManager.setSyncWindow`, `recordingCapMB` тЖТ `recorder.setCapMB`, `hubUrl` тЖТ `connection.store` initial URL (`''` = auto), display-unit prefs `angleUnit` (`deg`|`rad`) / `lengthUnit` (`m`|`mm`) read by components (values stay SI тАФ rad/m тАФ and are only converted for display, e.g. RobotModel's joint sliders + TCP nudge), and `demoMode` (a global toggle read by `RobotModelPlugin`, which **subscribes** to this store тАФ one-way dep, no cycle). Hydrates from localStorage (`webviz.settings`) and `apply()`s to the live singletons (and the DOM `data-theme`) at module load and on every `set()`; `persistedHubUrl()` is read by `connection.store` at init. **Does not** import `connection.store` (one-way dep to avoid a cycle). The themes are (mostly) pure CSS: `styles.css` defines the default ("sci-fi telemetry") palette as `:root` CSS variables and overrides them under `[data-theme='minimal']` / `[data-theme='vibrant']` / `[data-theme='industry']`, so switching only swaps variable values (glow/elevation vars flatten to `none` in the minimal/industry themes). `industry` is a light "engineering software" theme (├а la Visual Components/CAD) тАФ the first light theme, so it also flips `--surface-hi` to a dark overlay and `--accent-ink` (text on accent fills, e.g. primary buttons) to white. The 3D viewport isn't CSS-painted, so it tracks the theme via three viewport vars (`--viewport-bg` / `--grid-major` / `--grid-minor`) that `core/SceneManager.ts` reads (`applyTheme()`, called by `ThreeDTab` on theme change).
- `store/layouts.ts` тАФ named/shared layouts (┬з8 LayoutManager). `serializeWorkspace`/`applyWorkspace` convert between the live store and a `WorkspaceConfig` (`{version, tabs, split?, connection}`; `activeTabId` is still read off old pre-pivot configs but ignored; `applyWorkspace` runs `gcPanels` so orphaned pre-pivot background tabs are dropped); `listLayouts`/`saveLayout`/`loadLayout`/`deleteLayout` hit the hub's `/api/layouts` REST API (relative URL тЖТ Vite dev proxy or the hub-served prod build). The localStorage copy is the per-browser working state; the hub is the shared source of truth. Loading restores tabs but not the connection URL (so it won't yank a live session). The TopBar ЁЯТ╛ menu offers **New (fresh) layout** (`tabs.store.newWorkspace` тАФ reset to one empty pane), **Save currentтАж**, and per-layout load + тЬХ delete (New and delete both confirm first). тП║ record toggles `core/recorder.ts` (live REC indicator тЖТ `.wvrec` download on stop); ЁЯУВ loads a `.wvrec` for playback (`core/player.ts`); тЪЩ opens the settings popover (`store/settings.store.ts` тАФ theme, sync window, recording cap, default hub URL, reset).
- `ui/WorkspaceView.tsx` тАФ renders the workspace **always** as the `split` tree (`Node` тЖТ `Branch`/`Pane`), nested flex rows/columns sized by each branch's `frac`, with a `.split-divider` flex strip between siblings (pointer-captured drag тЖТ `setSplitFrac`, so it survives a pane's WebGL canvas). A lone leaf root is just one pane filling the area. Each `Pane` header has a panel-type `<select>` (тЖТ `setPaneType`), тЦе/тКЯ split buttons, a тдв/тдб maximize toggle (`toggleMaximize` тАФ the maximized pane floats `position:absolute` over its siblings, which stay mounted so their state survives a restore), and a тЬХ (`closePane`); an empty pane (no panel) shows a `PanelPicker` grid instead. The pane carries `data-tabtype` for the per-type accent. `App.tsx` renders `<WorkspaceView/>` (no `<TabBar/>`). All panes stay mounted (per-pane `SceneManager`/refs; shared `TFManager`/`HubClient`), so switching focus/maximize never tears a panel down.
- `tabs/TabRenderer.tsx` dispatches a `TabConfig` (the panel) to its renderer; all six are live (`InspectorTab`, `ThreeDTab`, `ImageTab`, `PlotTab`, `MapTab`, `LogTab`). `PlaceholderTab.tsx` now only serves unknown/custom panel types via the `default` case. Each panel is wrapped (in `WorkspaceView`) in `ui/TabErrorBoundary.tsx` тАФ a panel that throws (e.g. the 3D tab when the browser can't create a WebGL context) shows an inline error instead of unmounting the whole app, with a tailored hint for WebGL-context failures.
- **Image tab** (`tabs/ImageTab.tsx`) тАФ the ┬з11.3 camera grid and the first consumer of the binary data path *outside* the 3D scene. A user-configurable N├ЧM grid (1├Ч1/1├Ч2/2├Ч2/3├Ч2, persisted in tab settings as `{ layout, cells }`); each cell binds one `wv/Image` channel (dropdown filtered to that schema) and blits decoded frames to a `<canvas>` sized to the source image and scaled via CSS `object-fit: contain`. Decodes with the shared `decodeImagePayload`: JPEG/PNG via `createImageBitmap`, RGB8 via `ImageData`. Skips frames while an async decode is in flight (no backlog under a fast publisher).
- **Plot tab** (`tabs/PlotTab.tsx` + `core/plotSeries.ts`) тАФ the ┬з11.4 live time-series chart, organized as **subplots**: settings persist as `{ plots: [{ id, series: [{channel, field}] }], windowSec }`. Each subplot is its own coordinate system (independently auto-scaled y-axis) sharing the global x time-window тАФ so multiple fields in one subplot share an axis, while different channels in separate subplots get independent axes (a small signal isn't flattened by a large one). Old `{ series }` settings migrate to one subplot; new tabs start with one empty subplot. **Hand-rolled `<canvas>` per subplot with a coalesced rAF draw loop** (not Recharts, despite ┬з12) тАФ the window scrolls every frame, where a declarative chart's per-frame reconciliation would thrash; series are coloured by index *within* their subplot.
- **Map tab** (`tabs/MapTab.tsx`) тАФ the ┬з11.5 2D orthographic top-down view, a pure consumer of existing schemas (no new protocol): `wv/OccupancyGrid` base map + `wv/Path` polyline + `wv/LaserScan` points + a robot heading triangle from a TF frame. Everything is transformed into the shared **TFManager fixed frame** (`resolveToFixed`), reduced to a 2D affine `{tx, ty, yaw}` (`yawFrom*` extracts yaw from the quaternion) тАФ no three.js. The occupancy grid is decoded (base64 uint8) into an offscreen image once per message and placed with a canvas transform (worldтЖТscreen тИШ gridpixelтЖТworld via the origin pose); points use a JS `worldтЖТscreen`. Hand-rolled `<canvas>` + rAF; wheel zooms about the cursor (native non-passive listener), drag pans, auto-fit on first grid (and the тАЬтдв FitтАЭ button). Sidebar (`.mapd-side`) picks the fixed frame, per-layer channels (filtered by schema), the robot TF frame, grid alpha, and per-layer show toggles; a scale bar + тАЬN тЖСтАЭ are drawn in-canvas. World is +X right / +Y up (screen Y flipped).
- **Log tab** (`tabs/LogTab.tsx`) тАФ the ┬з11.7 event stream. **Aggregates every `wv/Log` channel** into one filtered, auto-scrolling list (like TFManager aggregates all transform channels) тАФ subscribes per distinct `wv/Log` channel and pushes into a rolling ref buffer (`MAX_ENTRIES`), rendering only the newest `MAX_RENDER` filtered rows. Filters persist in tab settings (`{ levels, search }`; DEBUG off by default); text search matches name+message. Pause freezes the view by skipping the 200 ms re-render tick (logs keep buffering, nothing dropped); Clear empties the buffer; the list sticks to the tail unless the user scrolls up. Levels are colour-coded (INFO green / WARN yellow / ERROR red, rows tinted). `plotSeries.ts` is the pure (node-testable) field layer: `discoverFields` lists a payload's numeric leaves as dot-paths, special-casing `wv/JointState` to one field per joint name; `readField` resolves a series' current value (joint-name lookup or dot-path). One subscription per distinct channel across all subplots; each frame is pushed once per distinct `(channel, field)` into a rolling buffer trimmed by window + `MAX_POINTS`; orphaned buffers are pruned when the series set changes. Live data lives in refs (canvases read them directly); the React-rendered legends (Hz / point counts) are kept fresh by a low-frequency forced re-render. Channel picker filters out binary (`encoding === 'binary'`) channels. **Pause freezes the time axis** (a shared `viewRef` `{end, span, latest}`; `null` = live and follows `now`) and makes the frozen view **zoom/pan-able** тАФ wheel zooms about the cursor (native non-passive listener), drag pans, both clamped to the retained data; buffers retain `RETENTION_SEC` (120 s, тЙл the view window) so there's scrollback to inspect; the y-axis auto-scales to the visible time slice. The view is shared across subplots so they stay x-aligned; тАЬтдв FitтАЭ resets the zoom.
- **3D tab** (`tabs/ThreeDTab.tsx`) тАФ three-column workspace (Displays sidebar ┬╖ Three.js viewport ┬╖ Properties panel). Per-tab `SceneManager`; the TF tree and hub connection are shared singletons. The Properties panel is the generic schema-form (`PropSchema`) for most plugins, but RobotModel gets a custom panel (`tabs/RobotModelProperties.tsx`).
- `core/SceneManager.ts` тАФ owns the three.js scene/camera/OrbitControls and a **coalesced** rAF loop (`requestRender()` marks dirty; idle viewport draws nothing). World is +Z up. Plugins add/remove `Object3D`s by id.
- `core/TFManager.ts` тАФ shared singleton TF tree: subscribes to all `wv/Transform`(`Array`) channels, 5s rolling buffer per frame, **SLERP/LERP interpolation**. `resolveToFixed(frame)` is **tf2-style**: if the fixed frame is an ancestor it composes straight up the parent chain; otherwise it walks *both* frames to their roots and combines via the shared root (`(T_root_fixed)тБ╗┬╣ тИШ T_root_frame`) тАФ so any frame connected through a common ancestor resolves (fixed frame needn't be a root), null only for disconnected trees/cycles. Also records the publishing channel per frame (`getFrameSource`/`getSourceChannels`) so displays can filter by source.
- `core/plugin.ts` тАФ `DisplayPlugin` contract + `PluginContext` (`{ hub, tf, scene }`) + `pluginRegistry`. `plugins/index.ts` registers built-ins (import for side effect before reading the catalogue).

**Display-plugin catalogue** тАФ see [`packages/app/src/plugins/README.md`](packages/app/src/plugins/README.md) for the full per-plugin detail (load pipelines, jog/IK, demo mode, gizmo internals). One-liners:

A **jog** toggle (arm-like robots only) spawns a shadow clone with a TCP gizmo and drag-IK тАФ **native** (in-browser Jacobian, client-side, one-shot "Send to robot") or **external** (publishes `wv/Pose`, drives from a solver's `wv/JointState`) backend. **Demo mode** (`settings.demoMode`) delegates to `RobotDemoController` + `core/demoExecutor.ts`, a client-side fake controller that fakes live-state and consumes the Send locally (no hub).
- `plugins/TFFramesPlugin.ts` тАФ visualizes the TF tree (axes + labels + parent links); a pure TFManager consumer with no channel binding of its own.
- `plugins/MarkerPlugin.ts` тАФ one `wv/Marker` channel тЖТ a marker store keyed by `namespace/id`, honoring the add/modify/delete lifecycle + `lifetime`; all geometry types incl. async `mesh` (updates in place on re-publish).

### `sdks/python` тАФ minimal client
The SDK package is `sdks/python/webviz/`; the runnable examples all live under `sdks/python/demos/`. Each demo that uses the SDK adds the package's parent dir to `sys.path` at import (a 1-line bootstrap) so it runs straight from the repo without an install.
- `webviz/client.py` тАФ two WS endpoints (mirror the binary header from `binary.ts`; require `pip install 'websockets>=11'` for the `websockets.sync` API). `webviz.Client` is the **source** side (`role=source`): advertises channels and sends JSON/binary frames. `webviz.Consumer` is the **client** side (`role=client`): the receive counterpart тАФ `subscribe(name, callback)` routes JSON `message` frames to per-channel-name callbacks, deferring the subscribe until the channel is advertised (mirrors the app's `HubClient`); a background daemon thread drains the socket and binary frames are ignored (the consume path is JSON-only). `Consumer` is what powers the ROS adapter's reverse bridge.
- `map_sim_demo.py` тАФ dependency-free demo that feeds the **Map, Log, and Inspector** tabs from one script (┬з11.5). A `MapSim` holds a random world as hidden ground truth (border walls + random rectangular obstacles, `--seed` for repeatability) and drives a dot robot on a collision-avoiding random walk (publishing `mobile_base_linkтЖТodom` `wv/Transform`). It **raycasts** a 360┬░ `wv/LaserScan` against the truth and **builds the published `wv/OccupancyGrid` from those scans** (cells traversedтЖТfree, hitтЖТoccupied, restтЖТunknown) so the map fills in SLAM-style as the robot explores; the trajectory is a `wv/Path` (`trail`). It also publishes a spread of extra channels off the same sim state (folded in when the old `demo_source.py` was retired): a `wv/Marker` sphere riding the robot, a noisy `wv/Pose` localization estimate (`pose_estimate`), `wv/Custom` `battery` telemetry, and a sim-flavored `wv/Log` nav stream (WARN on each obstacle turn, periodic DEBUG/ERROR). All HTTP `/api/inject`. Drive the Map tab with fixed frame `odom`, map=`map`, scan=`scan`, path=`trail`, robot=`mobile_base_link`. The body frame is `mobile_base_link` (not the generic `base_link`) so it can run **alongside** `robot_demo.py` without colliding in the shared TF tree тАФ both share the `odom` root, so both still resolve/render under the global fixed frame.
- `robot_demo.py` / `pointcloud_demo.py` / `image_demo.py` тАФ WS demos built on `webviz.Client` (so they need `websockets`; `client.py` uses the `websockets.sync` API, which requires **websockets тЙе 11**): `robot_demo.py` is the **hub-side twin of the app's demo-mode executor** тАФ a UR5 `wv/RobotModel` (URDF + meshes fetched online over CORS, no local assets) whose base TF is **static** (model + base TF advertised **latched**, published once тАФ the hub replays them to late-joining clients); it advertises `joint_states` as *feedback* and also opens a `webviz.Consumer` (`role=client`) subscribed to the jog "Send to robot" command channel (`ik/joint_states`), driving each joint toward the commanded goal at a velocity limit (`RobotController.step`) and streaming the motion тАФ so a real-robot native-IK loop (Jog тЖТ drag тЖТ Send) plays out on the hub (app setup: Joints ch.=`joint_states`, fixed frame=`base_link`; `--command-channel`/`--max-vel` override). `pointcloud_demo.py` streams animated binary `wv/PointCloud` frames; `image_demo.py` streams an animated RGB8 `wv/Image` (`camera_front`) for the Image tab. Each `*_payload()` duplicates the matching `binary.ts` payload layout.
- `ik_solver_demo.py` тАФ the **consume-side** demo (uses both `webviz.Consumer` *and* `Client`; needs `numpy` + `websockets`), and the reference implementation of RobotModel's **external** IK backend. Parses a URDF (default the bundled `demo_arm`) into a serial chain, subscribes to the gizmo's `tcp_target` (`wv/Pose`), solves damped-least-squares Jacobian IK (an `IkChain` mirroring `core/ik.ts` тАФ FK + geometric Jacobian + DLS, warm-started), and publishes `ik/joint_states` (`wv/JointState`). Meant as a template: swap `IkChain.solve` for MoveIt/KDL/ikfast. Assumes the target is in the robot base frame (set the app's fixed frame to the base link).

### `sdks/ros2` тАФ ROS 2 adapter (┬з6.2)
A **drop-in** `ament_python` package (also runnable straight from the repo) that bridges ROS topics into the hub with zero robot-code changes, built on `sdks/python`'s `webviz.Client`. The node (`webviz_ros2_adapter/adapter.py`) **polls `get_topic_names_and_types()`** and auto-subscribes to every topic whose ROS type appears in the `REGISTRY` (`registry.py`), advertising one wv channel per topic (name = topic minus leading slash). `converters.py` holds the **pure** ROSтЖТ`wv/*` functions тАФ it imports no `rclpy`/message packages (touches only message attributes), so it's node-testable with duck-typed stand-ins; binary converters (PointCloud2/Image/CompressedImage) return the post-header payload mirroring `binary.ts` (a **third** copy of the layout, alongside `client.py` and the demos). QoS is type-aware: sensor streams тЖТ best-effort `qos_profile_sensor_data`, `/tf_static` тЖТ transient-local (and advertised **latched** on the wv side, so its durability survives the bridge), else reliable depth-10. `--include`/`--exclude` regexes filter topics. To support a new ROS type, add a `TypeEntry` + converter.

The optional **reverse bridge** (`--enable-reverse`, off by default since it lets the browser command the ROS graph) carries the small *fixed* set of **interactive** messages WebViz itself can emit back onto ROS topics. It keys on the wv channel *name* (not ROS type тАФ the producer side is known and finite), driven by `registry.py`'s `REVERSE_REGISTRY` (a `ReverseEntry` per interactive channel): today the CoordinateFrame gizmo's `wv/Pose` (`tcp_target`) тЖТ `geometry_msgs/PoseStamped` on `/tcp_target`. The node opens a second `webviz.Consumer` (`role=client`) alongside its source `Client`, subscribes to each reverse channel, and on each frame builds + stamps + publishes the ROS message. Reverse converters live in `converters.py` as **`fill_*_from_wv(data, msg)`** functions that *mutate a passed-in ROS message* тАФ so the module still imports no message packages and stays node-testable (the node owns the classes). To prevent a ROSтЖТwvтЖТROS loop, reverse target topics are excluded from forward discovery. Override a topic with `--reverse-map CHANNEL=/topic`. Add a future interactive producer тЖТ add one `ReverseEntry` row.

### `sdks/cpp` тАФ C++ SDK (┬з6.3)
A **header-only**, dependency-free source client (`include/webviz/client.hpp`) тАФ POSIX/`ws://` only. It speaks just enough RFC 6455 over a raw TCP socket (own handshake + framing) to advertise channels and push frames, so it needs no Boost/OpenSSL/WS library; `#include` it and go. API mirrors `client.py`: `Client::advertise` тЖТ `Channel` with `send` (JSON, via a tiny `webviz::Value` builder тАФ objects `{{"k",v}}`, arrays `webviz::arr({...})`), `send_binary`, `send_image`, `send_pointcloud`. **Zero-copy binary path**: the small frame headers are built on the stack and the bulk payload is emitted with one scatter-gather `writev()` тАФ the camera/cloud buffer is never copied. To make that possible every client frame uses an **all-zero WebSocket mask key** (RFC 6455 needs the MASK bit, but a zero key is unmask-identity, avoiding a per-byte XOR copy; safe for a trusted LAN hub). A background thread drains inbound frames (answers ping, honors close). `test/test_payloads.cpp` asserts the encoded bytes match `binary.ts` (the cross-language contract; no network) and is a fourth copy of the payload layout. Build via CMake (`cmake -S . -B build && cmake --build build`; `ctest` runs the test); `examples/{transform,pointcloud}_demo.cpp` need a running hub.

## When changing the protocol

Edit `packages/protocol/src`, rebuild it, then update consumers. The Python SDK's binary header packing (`struct.pack("<B3xIdI", тАж)`) duplicates `binary.ts` тАФ keep both in sync when the frame layout changes; likewise `pointcloud_demo.py`'s `pointcloud_payload()`, the ROS 2 adapter's `converters.py` (`pointcloud2_to_payload`/`image_to_payload`), and the C++ SDK's `client.hpp` (`append_binary_header`/`image_payload_prefix`/`pointcloud_payload_prefix`, guarded by `sdks/cpp/test/test_payloads.cpp`) duplicate the `wv/PointCloud` / `wv/Image` payload layouts. `PROTOCOL_VERSION` lives in `protocol/src/index.ts` and is sent in the `server_info` handshake.
