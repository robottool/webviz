## What this is

WebViz is a browser-based visualization platform for robots and real-time systems. Three principles drive the design (`webviz_design_doc_v3.md` is the spec; code comments cite its sections as `§N`):

- **Protocol-first** — the `wv/*` wire protocol (`packages/protocol`) is the single contract shared by hub, app, and SDKs. Change it there and only there.
- **Source-agnostic** — the browser reasons only about channel *names* + schema types, never about where data came from (ROS, file, custom SDK). The hub erases origin.
- **Tiling-panel workspace** — the app is a recursive split layout of independent panels sharing one connection (no tab bar; split to add a panel, close to remove, maximize to focus). "Switch between whole setups" is named layouts, not tabs.

The repo is a **vertical slice in progress**. **All six tabs are live** — **Inspector**, **3D**, **Image**, **Plot**, **Map**, and **Log** — and the 3D display-plugin catalogue is essentially complete: `RobotModel`, `TFFrames`, `Marker`, `PointCloud`, `LaserScan`, `OccupancyGrid`, `Path`, `Pose`, `CoordinateFrame` (plus TFManager and SceneManager). Recording **capture *and* playback** both work now (`.wvrec` capture + replay-as-a-source). The **Python SDK**, a **ROS 2 adapter** (`sdks/ros2`), and a header-only **C++ SDK** (`sdks/cpp`, §6.3) all exist — the design doc's SDK catalogue is complete.

## Commands

Toolchain note: pnpm 9 runs via corepack from `~/.local/bin` (Node 20 quirk) — ensure that's on `PATH` before running pnpm.

```bash
./setup.sh                                 # idempotent first-time Linux setup: installs toolchain (Node ≥22 via nvm if missing, pnpm), JS deps, builds protocol, and a Python venv with websockets for the demos. Re-run safely; only fills in what's missing.
./dev.sh                                   # one-shot local stack: builds protocol, then runs hub + app together (binds 0.0.0.0; tears both down on Ctrl+C). Header comments cover VM/remote port-forwarding.

pnpm install                              # install workspace deps
pnpm --filter @webviz/protocol build      # MUST build protocol first — hub & app consume its dist/
pnpm build                                 # build all packages (pnpm -r)
pnpm typecheck                             # typecheck all packages

pnpm hub                                   # run hub: WS broker :7777 + HTTP/asset :8080 (tsx watch)
pnpm app                                   # run app dev server (Vite) :5173, auto-connects ws://localhost:7777
python3 sdks/python/demos/demo_source.py         # feed demo data via HTTP /api/inject (no pip deps)
python3 sdks/python/demos/map_sim_demo.py        # feed the Map tab: scan-built map + wandering robot + raycast scan (HTTP, no pip deps)
# The WS demos below need websockets>=11; ./setup.sh provisions ./venv with it, so run them via venv/bin/python3 (or any env that has websockets):
venv/bin/python3 sdks/python/demos/robot_demo.py          # feed the 3D tab: UR5 RobotModel + JointState + base TF
venv/bin/python3 sdks/python/demos/pointcloud_demo.py     # feed the 3D tab: animated binary wv/PointCloud
venv/bin/python3 sdks/python/demos/image_demo.py          # feed the Image tab: animated binary wv/Image RGB8
venv/bin/python3 sdks/python/demos/ik_solver_demo.py      # external IK backend: solve tcp_target → ik/joint_states (needs numpy; ./setup.sh provisions it)
```

The hub's asset server defaults `assetsDir` to the **repo root**, so any bundled assets are reachable at `/assets/<dir>/...` out of the box (override with `WEBVIZ_ASSETS_DIR` in production). The `RobotModel` plugin can fetch URDF + meshes from `http://<host>:8080/assets`, but the repo no longer ships a robot description — the `robot_demo.py` UR5 and the `demo_source.py` mesh marker load their URDF/meshes **online over CORS** from the upstream `Gepetto/example-robot-data` repo (`raw.githubusercontent.com`) instead.

Tests (vitest, only in `@webviz/protocol` — two files co-located in `src/`: `binary.test.ts` and `frame.test.ts`; note the C++ SDK instead uses a `test/` dir):
```bash
pnpm test                                            # run the whole workspace's tests (pnpm -r test; today only protocol's)
pnpm --filter @webviz/protocol test                 # run protocol's once
pnpm --filter @webviz/protocol test:watch           # watch
pnpm --filter @webviz/protocol exec vitest run binary   # single test file by name (or: frame)
```

`@webviz/protocol` must be built before hub/app typecheck or build, because they import from its `dist/` via `workspace:*`. The hub uses `tsx watch` in dev so it does not need a prior build for `pnpm hub`.

**Static (hub-less) demo build.** `WEBVIZ_BASE=/<repo>/ pnpm --filter @webviz/app build` emits a static bundle whose asset URLs are rooted at `/<repo>/` for GitHub Pages (`.github/workflows/pages.yml` deploys it on push to `main`). `base` defaults to `/` (env-gated in `vite.config.ts`), so the hub-served build and dev server are unchanged. On the static deploy only the fully client-side features work — loading a URDF (including the bundled **demo robot**, `RobotModelProperties` → "load demo robot") and driving it with the manual joint sliders; everything hub-backed (live channels, demos, playback, layouts) is unavailable.

## Architecture

Three TS packages + a Python SDK, pnpm workspace (`packages/*`), ESM throughout (`"type": "module"`), strict TS. Note that intra-package imports use `.js` extensions (ESM/`moduleResolution: Bundler`) even though sources are `.ts`.

### `packages/protocol` — the wire contract
- `schemas.ts` — `wv/*` schema types (Transform, Marker, Image, PointCloud, …): the shape of each channel's `data` payload.
- `messages.ts` — JSON control/data ops over the WS text channel (`server_info`, `advertise`, `subscribe`, `message`, …), with discriminated-union types (`ClientMessage`, `SourceMessage`, `ServerMessage`, `AnyMessage`) for exhaustive routing.
- `binary.ts` — binary data frames. **Fixed 20-byte little-endian header**: `op(0x01)` / 3 reserved / `uint32 channel_id` / `float64 timestamp` / `uint32 payload_length` / payload. Little-endian is a WebViz decision (the spec doesn't mandate one) — keep encoder and decoder in sync. Also holds the schema-specific binary payload codecs: `wv/Image` and `wv/PointCloud` both lead with a length-prefixed `frame_id` (so the cloud/image carries its own TF frame); the PointCloud float region is generally unaligned, so the decoder copies it into an aligned buffer before viewing it as `Float32Array`.
- `frame.ts` — `decodeFrame(raw)` normalizes any incoming WS frame (string JSON or binary) into a `data` or `control` result. The one decode entry point used by both hub and app.

### `packages/hub` — broker + asset server (`server.ts`'s `startHub()` wires both, sharing one registry)
- `server.ts` — `startHub(opts)`: the programmatic entry point. Constructs the `Broker` + asset server and resolves a `RunningHub` (`{wsPort, httpPort, close()}`) once HTTP is listening (rejects if a port can't bind). Keeping the wiring here — not at module top-level — lets a host embed the hub in-process. `main.ts` is the CLI shell that calls `startHub` with env-overridable ports/paths (`WEBVIZ_*`) and logs; the hub `package.json` `exports` `.` → `dist/server.js` so importers get `startHub` while `bin` still points at `main.js`. `Broker.close()` tears down the WS server for clean embedded shutdown.
- `broker.ts` — WebSocketServer on :7777. Same port serves both roles, distinguished by query string `?role=source|client` (`&id=<sourceId>`). Sources advertise channels and push frames; the broker fans each frame out to subscribed clients, enforcing per-`(channel, client)` `max_hz` throttling. `injectJson()` lets a non-WS HTTP caller publish (used by `/api/inject`).
- `channel_registry.ts` — maps each source's *local* channel ids to *global* ids, renaming on collision (`resolveLocal`, `advertise`). This is the source-agnostic erasure layer.
- `session_store.ts` — `SessionStore`: persists workspace layouts as JSON files under `dataDir` (`WEBVIZ_DATA_DIR`, default `data/layouts`), backing the `/api/layouts` REST endpoints. The hub is the shared/multi-user source of truth; the app also keeps a localStorage copy. Layout names are sanitized to a flat, safe filename namespace.
- `asset_server.ts` — HTTP :8080: serves the built app, plus REST `GET /api/channels`, `GET/POST /api/layouts` (+ `/:id`), `POST /api/inject`, and `/assets/*`. Has path-traversal guarding for static serving. The `/assets/` prefix is shared by two roots — robot descriptions/meshes (`assetsDir`) and the app's own Vite bundle (which Vite also emits under `/assets/`, i.e. `webDir/assets/*`) — so it resolves `assetsDir` first, then falls back to the app bundle, then 404s.
- Ports/dirs are overridable via env: `WEBVIZ_WS_PORT`, `WEBVIZ_HTTP_PORT`, `WEBVIZ_WEB_DIR`, `WEBVIZ_ASSETS_DIR`, `WEBVIZ_DATA_DIR`, `ALLOWED_ORIGINS`.

### `packages/app` — Vite + React + Zustand + three.js
The data path is **HubClient → TimeManager → MessageRouter → tab handlers**:
- `core/TimeManager.ts` — the §8 sync-window reorder buffer between the socket and the router. `HubClient` enqueues each decoded data frame; the TimeManager holds it for a small window (default 20 ms) and flushes in **source-timestamp order**, so a `wv/Transform` published alongside a sensor frame is dispatched (and TF-indexed) first. Ordering keys on the source clock (`maxSeenT`), never browser wall-clock (skew-safe in a VM); a wall-clock `maxHold` only force-flushes frames whose channel went quiet. `getCurrentTime()` exposes the latest source time.
- `core/recorder.ts` — session recorder (§16.1). Taps the **raw** WS frames at the top of `HubClient.onMessage` (before decode), so a recording is byte-faithful, including binary `wv/PointCloud`/`wv/Image` frames. Records only what the browser actually receives (i.e. subscribed channels). `start(channels)` snapshots the live channel list as a **catalogue** (data frames carry only a numeric `channel_id`; the `advertise`/`server_info` that *name* a channel arrive at connect, before recording — without the snapshot a mid-session recording couldn't be replayed under meaningful names). `stop()` serializes a `.wvrec` container (`WVR2` magic, then `uint32 catalogueLen` + JSON `ChannelInfo[]`, then per record `float64 t` / `uint8 type` / `uint32 len` / payload) as a Blob; the TopBar ⏺ button downloads it. The reader still accepts legacy `WVR1` (no catalogue). Recordings are held in memory, so `capture()` enforces a **cap** (`MAX_BYTES` 256 MB / `MAX_FRAMES` 200k, whichever first): on hit it freezes the buffer and sets `isLimitReached()`, and the `RecordButton` poll auto-finalizes the download. The cap bounds memory and the cost of a backward seek in playback (which replays from the start — kept lossless rather than per-channel-last-value, because accumulating channels like Markers' add/delete lifecycle and the Log stream would break).
- `core/player.ts` — recording **playback** (§16.1), singleton `player`. Replays a `.wvrec` **as a second hub source** (`ws://<host>:7777?role=source&id=replay`) so it **coexists with a live connection** rather than replacing it. The trick: advertise each recorded channel using its *original recorded global id* as the source's **local** id, so the recorded frames replay **byte-for-byte unchanged** — the hub remaps `(connId, localId) → fresh global id` and rewrites JSON `channel_id` / binary headers on the way out (`channel_registry.ts`). Replayed channels appear to the live `HubClient` as `replay/<name>`. Parses both `WVR2` (catalogue from the header) and `WVR1` (catalogue rebuilt from in-stream `advertise` frames); only `op:'message'` + binary data frames are replayed (control frames skipped). A rAF clock advances by `realΔ × speed` and flushes due records in timestamp order; transport API is `play/pause/seek/setSpeed/unload` (backward seek replays from t=0 so latest-value channels rebuild; `unload` closes the source socket → hub unadvertises the `replay/*` channels). `store/playback.store.ts` mirrors its state into React (like `connection.store`); `ui/PlaybackBar.tsx` is the bottom transport bar (▶/⏸, scrubber, speed, ✕), rendered in `App.tsx` only when loaded; the TopBar 📂 button loads a file.
- `protocol/HubClient.ts` — **singleton** (`hubClient`), one WebSocket per browser window. Owns reconnection, the live channel list (from `server_info`/`advertise`/`unadvertise`), and **reference-counted subscriptions by channel name**: first subscriber sends `subscribe`, last sender of unsubscribe sends `unsubscribe`. Subscribing to a not-yet-advertised channel is deferred until it appears. Subscribe by *name*, not id.
- `protocol/MessageRouter.ts` — fans one decoded frame out to every registered per-channel-id handler, so multiple tabs on the same channel each get their own callback.
- `store/connection.store.ts`, `store/tabs.store.ts` — Zustand stores for connection state and the **panel workspace**. `TAB_META` defines the six panel types. `tabs.store` **persists the workspace to localStorage** (debounced) and hydrates on load. `connection.store`'s initial URL is `persistedHubUrl() || <auto-derived>` (settings override wins). The workspace is a **tiling layout** (`SplitState = {root, focusedLeaf, maximized}`), a **recursive binary tree** (`SplitNode` = `leaf{id,tabId}` | `split{id,dir:'row'|'col',frac,a,b}`). Each leaf pane *owns* one panel; `tabs` is the panel pool (a `TabConfig` per leaf), keyed by the leaf's `tabId` (null = empty pane awaiting a pick). Splitting replaces a leaf with `[old leaf, new **empty** leaf]`; closing collapses the parent to the sibling (closing the only pane clears it to one empty pane). Pure tree helpers (`splitLeafNode`/`removeLeafNode`/`mapLeaves`/`setFracNode`/`leafIds`/`findLeaf`) back the actions `splitPane(leafId,dir)`, `closePane(leafId)`, `setPaneType(leafId,type)` (the "pick content" action — mints a panel, assigns it), `setSplitFrac(branchId,frac)`, `focusPane`, `toggleMaximize(leafId)`, plus `updateSettings`/`newWorkspace`. `gcPanels` (exported) drops panels no pane owns any more so the pool can't leak orphans; it runs after every structural change and on load. `sanitizeSplit(raw, tabs)` (exported) validates a persisted/loaded tree (drops stale tab ids→null, clamps fracs, regenerates missing node ids, validates `maximized`); no usable tree → one pane bound to the first panel (or empty). There is **no tab bar** (the old `ui/TabBar.tsx` and `addTab`/`activateTab`/`activeTabId` were removed in the panel pivot).
- `store/settings.store.ts` — the few **global** knobs the ⚙ panel exposes (most config is per-tab/per-display, not here): `theme` (`'telemetry'`|`'minimal'`|`'vibrant'`|`'industry'`, default `telemetry`) → `document.documentElement.dataset.theme`, `syncWindowMs` → `TimeManager.setSyncWindow`, `recordingCapMB` → `recorder.setCapMB`, `hubUrl` → `connection.store` initial URL (`''` = auto). Hydrates from localStorage (`webviz.settings`) and `apply()`s to the live singletons (and the DOM `data-theme`) at module load and on every `set()`; `persistedHubUrl()` is read by `connection.store` at init. **Does not** import `connection.store` (one-way dep to avoid a cycle). The themes are (mostly) pure CSS: `styles.css` defines the default ("sci-fi telemetry") palette as `:root` CSS variables and overrides them under `[data-theme='minimal']` / `[data-theme='vibrant']` / `[data-theme='industry']`, so switching only swaps variable values (glow/elevation vars flatten to `none` in the minimal/industry themes). `industry` is a light "engineering software" theme (à la Visual Components/CAD) — the first light theme, so it also flips `--surface-hi` to a dark overlay and `--accent-ink` (text on accent fills, e.g. primary buttons) to white. The 3D viewport isn't CSS-painted, so it tracks the theme via three viewport vars (`--viewport-bg` / `--grid-major` / `--grid-minor`) that `core/SceneManager.ts` reads (`applyTheme()`, called by `ThreeDTab` on theme change).
- `store/layouts.ts` — named/shared layouts (§8 LayoutManager). `serializeWorkspace`/`applyWorkspace` convert between the live store and a `WorkspaceConfig` (`{version, tabs, split?, connection}`; `activeTabId` is still read off old pre-pivot configs but ignored; `applyWorkspace` runs `gcPanels` so orphaned pre-pivot background tabs are dropped); `listLayouts`/`saveLayout`/`loadLayout`/`deleteLayout` hit the hub's `/api/layouts` REST API (relative URL → Vite dev proxy or the hub-served prod build). The localStorage copy is the per-browser working state; the hub is the shared source of truth. Loading restores tabs but not the connection URL (so it won't yank a live session). The TopBar 💾 menu offers **New (fresh) layout** (`tabs.store.newWorkspace` — reset to one empty pane), **Save current…**, and per-layout load + ✕ delete (New and delete both confirm first). ⏺ record toggles `core/recorder.ts` (live REC indicator → `.wvrec` download on stop); 📂 loads a `.wvrec` for playback (`core/player.ts`); ⚙ opens the settings popover (`store/settings.store.ts` — theme, sync window, recording cap, default hub URL, reset).
- `ui/WorkspaceView.tsx` — renders the workspace **always** as the `split` tree (`Node` → `Branch`/`Pane`), nested flex rows/columns sized by each branch's `frac`, with a `.split-divider` flex strip between siblings (pointer-captured drag → `setSplitFrac`, so it survives a pane's WebGL canvas). A lone leaf root is just one pane filling the area. Each `Pane` header has a panel-type `<select>` (→ `setPaneType`), ▥/⊟ split buttons, a ⤢/⤡ maximize toggle (`toggleMaximize` — the maximized pane floats `position:absolute` over its siblings, which stay mounted so their state survives a restore), and a ✕ (`closePane`); an empty pane (no panel) shows a `PanelPicker` grid instead. The pane carries `data-tabtype` for the per-type accent. `App.tsx` renders `<WorkspaceView/>` (no `<TabBar/>`). All panes stay mounted (per-pane `SceneManager`/refs; shared `TFManager`/`HubClient`), so switching focus/maximize never tears a panel down.
- `tabs/TabRenderer.tsx` dispatches a `TabConfig` (the panel) to its renderer; all six are live (`InspectorTab`, `ThreeDTab`, `ImageTab`, `PlotTab`, `MapTab`, `LogTab`). `PlaceholderTab.tsx` now only serves unknown/custom panel types via the `default` case. Each panel is wrapped (in `WorkspaceView`) in `ui/TabErrorBoundary.tsx` — a panel that throws (e.g. the 3D tab when the browser can't create a WebGL context) shows an inline error instead of unmounting the whole app, with a tailored hint for WebGL-context failures.
- **Image tab** (`tabs/ImageTab.tsx`) — the §11.3 camera grid and the first consumer of the binary data path *outside* the 3D scene. A user-configurable N×M grid (1×1/1×2/2×2/3×2, persisted in tab settings as `{ layout, cells }`); each cell binds one `wv/Image` channel (dropdown filtered to that schema) and blits decoded frames to a `<canvas>` sized to the source image and scaled via CSS `object-fit: contain`. Decodes with the shared `decodeImagePayload`: JPEG/PNG via `createImageBitmap`, RGB8 via `ImageData`. Skips frames while an async decode is in flight (no backlog under a fast publisher).
- **Plot tab** (`tabs/PlotTab.tsx` + `core/plotSeries.ts`) — the §11.4 live time-series chart, organized as **subplots**: settings persist as `{ plots: [{ id, series: [{channel, field}] }], windowSec }`. Each subplot is its own coordinate system (independently auto-scaled y-axis) sharing the global x time-window — so multiple fields in one subplot share an axis, while different channels in separate subplots get independent axes (a small signal isn't flattened by a large one). Old `{ series }` settings migrate to one subplot; new tabs start with one empty subplot. **Hand-rolled `<canvas>` per subplot with a coalesced rAF draw loop** (not Recharts, despite §12) — the window scrolls every frame, where a declarative chart's per-frame reconciliation would thrash; series are coloured by index *within* their subplot.
- **Map tab** (`tabs/MapTab.tsx`) — the §11.5 2D orthographic top-down view, a pure consumer of existing schemas (no new protocol): `wv/OccupancyGrid` base map + `wv/Path` polyline + `wv/LaserScan` points + a robot heading triangle from a TF frame. Everything is transformed into the shared **TFManager fixed frame** (`resolveToFixed`), reduced to a 2D affine `{tx, ty, yaw}` (`yawFrom*` extracts yaw from the quaternion) — no three.js. The occupancy grid is decoded (base64 uint8) into an offscreen image once per message and placed with a canvas transform (world→screen ∘ gridpixel→world via the origin pose); points use a JS `world→screen`. Hand-rolled `<canvas>` + rAF; wheel zooms about the cursor (native non-passive listener), drag pans, auto-fit on first grid (and the “⤢ Fit” button). Sidebar (`.mapd-side`) picks the fixed frame, per-layer channels (filtered by schema), the robot TF frame, grid alpha, and per-layer show toggles; a scale bar + “N ↑” are drawn in-canvas. World is +X right / +Y up (screen Y flipped).
- **Log tab** (`tabs/LogTab.tsx`) — the §11.7 event stream. **Aggregates every `wv/Log` channel** into one filtered, auto-scrolling list (like TFManager aggregates all transform channels) — subscribes per distinct `wv/Log` channel and pushes into a rolling ref buffer (`MAX_ENTRIES`), rendering only the newest `MAX_RENDER` filtered rows. Filters persist in tab settings (`{ levels, search }`; DEBUG off by default); text search matches name+message. Pause freezes the view by skipping the 200 ms re-render tick (logs keep buffering, nothing dropped); Clear empties the buffer; the list sticks to the tail unless the user scrolls up. Levels are colour-coded (INFO green / WARN yellow / ERROR red, rows tinted). `plotSeries.ts` is the pure (node-testable) field layer: `discoverFields` lists a payload's numeric leaves as dot-paths, special-casing `wv/JointState` to one field per joint name; `readField` resolves a series' current value (joint-name lookup or dot-path). One subscription per distinct channel across all subplots; each frame is pushed once per distinct `(channel, field)` into a rolling buffer trimmed by window + `MAX_POINTS`; orphaned buffers are pruned when the series set changes. Live data lives in refs (canvases read them directly); the React-rendered legends (Hz / point counts) are kept fresh by a low-frequency forced re-render. Channel picker filters out binary (`encoding === 'binary'`) channels. **Pause freezes the time axis** (a shared `viewRef` `{end, span, latest}`; `null` = live and follows `now`) and makes the frozen view **zoom/pan-able** — wheel zooms about the cursor (native non-passive listener), drag pans, both clamped to the retained data; buffers retain `RETENTION_SEC` (120 s, ≫ the view window) so there's scrollback to inspect; the y-axis auto-scales to the visible time slice. The view is shared across subplots so they stay x-aligned; “⤢ Fit” resets the zoom.
- **3D tab** (`tabs/ThreeDTab.tsx`) — three-column workspace (Displays sidebar · Three.js viewport · Properties panel). Per-tab `SceneManager`; the TF tree and hub connection are shared singletons. The Properties panel is the generic schema-form (`PropSchema`) for most plugins, but RobotModel gets a custom panel (`tabs/RobotModelProperties.tsx`).
- `core/SceneManager.ts` — owns the three.js scene/camera/OrbitControls and a **coalesced** rAF loop (`requestRender()` marks dirty; idle viewport draws nothing). World is +Z up. Plugins add/remove `Object3D`s by id.
- `core/TFManager.ts` — shared singleton TF tree: subscribes to all `wv/Transform`(`Array`) channels, 5s rolling buffer per frame, **SLERP/LERP interpolation**, `resolveToFixed(frame)` composes the parent chain into the fixed frame. Also records the publishing channel per frame (`getFrameSource`/`getSourceChannels`) so displays can filter by source.
- `core/plugin.ts` — `DisplayPlugin` contract + `PluginContext` (`{ hub, tf, scene }`) + `pluginRegistry`. `plugins/index.ts` registers built-ins (import for side effect before reading the catalogue).
- `plugins/RobotModelPlugin.ts` — renders a URDF via `urdf-loader` with three independently switchable input sources: **URDF** (`local` folder upload **or** a `wv/RobotModel` channel), **joints** (`manual` sliders **or** `wv/JointState`), **pose** (`manual` xyz/rpy **or** TF/`resolveToFixed`). Defaults to all-channel so a published robot auto-displays; loading a local folder flips joints/pose to manual preview. Emits a validation report (joints + limits, meshes loaded/failed) consumed by `RobotModelProperties`. Channel-published meshes load cross-origin when the `wv/RobotModel`'s `urdf_url` is an absolute http(s) URL (`loadFromChannel` sets `remoteUrdfBase = repoBaseOf(url)`, resolving `package://` mesh refs against the repo base — same as a URL load), and from the hub asset server (anchored on `ur_description/`) for a relative/hub-served `urdf_url`; local meshes resolve to blob URLs. `loadFromManifest(baseUrl, paths)` is a third local entry point that **fetches** a file set over relative HTTP (used by the Properties "load demo robot" button, which loads the bundled primitive-geometry `packages/app/public/demo-robot/demo_arm.urdf`) — it converges on the same `reloadLocal` pipeline as the picker, so `localFiles` is `Array<File | PickedFile>` and access goes through `pathOfLocal`/`textOfLocal`. `loadFromUrdfUrl(input)` is a fourth: it loads a URDF from a **GitHub/raw URL** (`toRawUrl` normalizes `github.com/blob` → `raw.githubusercontent.com`; `repoBaseOf` derives the repo base; `resolveRepoMeshUrl` maps the URDF's `package://<pkg>/<rest>` mesh refs to `<base><rest>`, the common `*_description` layout) — `urdf-loader` then fetches each mesh on demand cross-origin, so it only works for CORS-enabled hosts and flat (non-xacro) URDFs. Mesh resolution in `loadMesh` is three-way: local blob resolver, else `remoteUrdfBase` (URL load **or** a channel load whose `urdf_url` is absolute http(s)), else the hub asset server (relative/hub-served channel load). This whole local path is fully client-side (no hub), so it's the one feature that works on a hub-less static deploy. The Properties "Load URDF…" button opens a dialog with all three sources (folder · URL · demo) and surfaces load errors. Joints have a **third source beyond channel/manual: `ik`** (`JointSource = Source | 'ik'`) — a client-side **drag-the-TCP** mode. Switching to it snapshots the current pose as the seed, **freezes the base**, and drops a move+rotate gizmo on a chosen TCP link; dragging it runs a damped-least-squares Jacobian solve (`plugins/RobotIkController.ts` + `core/ik.ts`) and re-poses the arm to follow. The controller (`plugins/RobotIkController.ts`) builds the actuated chain root→TCP by walking `urdf-loader`'s scene-graph parents, and owns the gizmo; while `joint_source === 'ik'` the plugin's `onRender` early-returns so the controller owns the joints and the base stays frozen. It has **two interchangeable solver backends** (`ik_backend`): **`native`** solves in-browser — FK from each joint's `matrixWorld`, an analytic geometric Jacobian from its world axis/origin, the DLS step from `core/ik.ts`, applied via `robot.setJointValue` (fully client-side, no hub) — the drag is a **preview** that publishes nothing, plus an on-demand **"Send pose to robot"** button (`RobotIkController.sendTarget()`) that publishes the current TCP pose *once* as a `wv/Pose` on `ik_target_channel` and **holds** it (a keepalive re-asserts that snapshot, so dragging afterwards doesn't move the robot until the next send); **`external`** publishes the gizmo pose as a `wv/Pose` target (`ik_target_channel`, default `tcp_target`) via `core/sourcePublisher.ts` (rate-capped drag + a `setInterval` keepalive so a late solver latches) and drives the robot from a `wv/JointState` **solution** channel (`ik_solution_channel`, default `ik/joint_states`) the user's own solver (MoveIt/KDL/ikfast/…) publishes back — so the drag-TCP UX stays but the kinematics are theirs (needs a hub). Either backend reports a reach residual (FK-vs-target); external shows "waiting for solver" until the first solution. The `ik` option is only offered for **arm-like** robots: `isIkFeasible()` (cached at load) requires a serial chain with ≥ `MIN_IK_DOF` (2) actuated joints, so a mobile base / static / single-joint model hides it (and a persisted `ik` source falls back to `channel` on loading an infeasible robot). Properties (`RobotModelProperties.tsx`'s `IkPanel`) exposes the TCP-link picker, the solver toggle (native: orientation-weight slider · external: target/solution channel inputs), the reach/residual readout, and "Recenter on current pose".
- `plugins/TFFramesPlugin.ts` — visualizes the TF tree: an `AxesHelper` triad + camera-facing canvas-texture label per frame, plus parent→child line segments. A **pure TFManager consumer** — no channel binding of its own; each frame is placed via `resolveToFixed` (the scene root *is* the fixed frame), unresolved frames are hidden. Uses the generic `PropSchema` Properties form (source filter, frame-name filter, axis scale, labels, parent links); a link is drawn only when both endpoints pass the filter.
- `plugins/MarkerPlugin.ts` — subscribes to one `wv/Marker` channel and keeps a marker **store** keyed by `namespace/id`, honoring the action lifecycle (`add`/`modify`/`delete`/`delete_namespace`/`delete_all`) and `lifetime` expiry. Each marker is a group anchored to its `frame_id` via `resolveToFixed` each render, with the geometry offset inside by the marker's own pose. Geometry: cube, sphere, cylinder, arrow, line_strip, line_list, points, triangle_list, and `mesh` (loaded asynchronously from the hub asset server via `core/meshLoad.ts`). The empty group/entry is created synchronously so the lifecycle tracks the identity, and the loaded object is attached only if the entry is still the live one for its key — guarding the replace/delete-mid-load race. A re-published mesh marker with the **same** `mesh_url` updates in place (pose/scale/color) rather than refetching, so a high-rate (e.g. 20 Hz) stream doesn't reload the resource every frame; `scale` multiplies the mesh's intrinsic `baseScale` (captured at load, so updates don't compound) and color is only re-tinted when it changes (`colorKey`). A non-zero marker `color` tints the whole mesh, an all-zero color keeps the mesh's embedded materials. `text` (needs a CSS2D renderer) is still deferred and ignored.
- `plugins/PointCloudPlugin.ts` + `plugins/pointcloud.worker.ts` + `core/pointcloudDecode.ts` — the performance-critical plugin (§10). Binary `wv/PointCloud` frames are **decoded in a WebWorker** (`pointcloud.worker.ts`, instantiated via Vite's `new Worker(new URL(...), { type: 'module' })`); the pure deinterleave lives in `pointcloudDecode.ts` (no three/Worker globals, so it's node-testable and shared by both). The plugin copies each payload out of the shared frame buffer, transfers it to the worker, and on the result swaps the buffers into a BufferGeometry. Coloring: RGB → vertex colors; intensity → viridis colormap (`core/colormap.ts` DataTexture, sampled in a `ShaderMaterial`) normalized per frame; xyz-only → colormap over z. Anchored to the payload's `frame_id` via `resolveToFixed`.
- `plugins/{LaserScan,OccupancyGrid,Path,Pose}Plugin.ts` — the JSON sensor/nav plugins, each subscribing to one channel of its schema and anchoring to `frame_id` via `resolveToFixed`. **LaserScan**: polar→Cartesian `THREE.Points` (drops out-of-range / `"Inf"` beams). **OccupancyGrid**: base64 uint8 → `DataTexture` (free=white, occupied=black, unknown=transparent) on a `PlaneGeometry` sized `w·h·resolution`, nested root(frame)▸origin(grid origin)▸plane. **Path**: `THREE.Line` polyline colored by the path's RGBA. **Pose**: an `ArrowHelper` along +X plus an optional 2σ covariance ellipse (`ellipse2D` does the 2×2 eigen of the position block).
- `plugins/CoordinateFramePlugin.ts` — the one **interactive** display (a *producer*, not a consumer). Shows a combined move+rotate gizmo the user manipulates to author a 6-DOF pose — e.g. an IK / TCP target — and **publishes** it as a `wv/Pose` channel (default name `tcp_target`) back to the hub via `core/sourcePublisher.ts`. three.js `TransformControls` (same `three/examples/jsm/controls` dir as `OrbitControls`) renders one mode at a time, so to show translate axes *and* rotate rings together it attaches **two** TransformControls to the same `gizmoNode` — one `translate`, one `rotate`; there is no separate triad mesh, the gizmo handles *are* the visual. The **rotate control is constructed first on purpose**: each TransformControls grabs pointer capture in listener (construction) order and the `dragging-changed` handler disables the *other* control while one drags, so the first-built wins any handle overlap — rotation gets priority (the move arrows are still grabbable on their outer, ring-free segments). `pruneGizmo` strips handles from each control's internal `_gizmo` (both the visible gizmo and the raycast picker): translate loses the `XY`/`YZ`/`XZ` plane rectangles — **removing their large pickers is what lets clicks reach the rotate rings** (otherwise the plane pickers swallow them and rotation appears dead) — and rotate loses the big outer `XYZE`/`E` rings, leaving just the X/Y/Z axis rings. The node lives under the scene root (= the fixed frame, at world identity), so its transform is the fixed-frame pose, published as the Pose's `frame_id` (`scene.getFixedFrame()`). On `dragging-changed` it freezes `OrbitControls` **and the other gizmo** so they don't fight; `objectChange` syncs the pose into settings (so the generic Properties form's X/Y/Z + roll/pitch/yaw fields reflect drags on its next render — and typing those fields drives the gizmo back). `onRender` publishes rate-capped while dragging (`PUBLISH_HZ`) with a low-rate keepalive (`KEEPALIVE_MS`) so late subscribers latch the current pose. the gizmo is always **local space** (`tc.setSpace('local')`) so the handles follow the frame's orientation — rotating a ring visibly turns the move arrows with it; gizmo `size` is a setting and the rotate rings are drawn at `RING_SCALE×` the move-arrow size.
- `core/sourcePublisher.ts` — lets the app publish data *back* to the hub as a second source (`role=source&id=ui`), the same coexists-with-a-live-connection trick as `core/player.ts`. The app's `HubClient` is a consumer (`role=client`) and can't advertise, so this singleton owns one shared source socket per window; `advertise(name, schema)` returns a `PublishHandle` with `send`/`close`. The hub remaps `(conn, localId)` → fresh global id and rewrites the frame `channel_id` (`channel_registry.ts`), so subscribers — including our own `HubClient` — see a new named channel. Each channel's last value is **latched** and re-sent on (re)connect. Currently the lone producer is `CoordinateFramePlugin`.
- `core/meshResolver.ts` — `LocalAssetResolver`: indexes folder-picked files by `webkitRelativePath`, resolves `package://`/relative refs by **longest trailing-segment match**, and exposes a `THREE.LoadingManager` whose `setURLModifier` routes every loader request (mesh + DAE textures / GLTF `.bin`) to a blob URL.
- `core/meshLoad.ts` — shared mesh loading for `RobotModelPlugin` and `MarkerPlugin`'s `mesh` type: `loadMeshFromUrl(url, format, material?, manager?)` dispatches to the right three.js loader by extension (`.dae`/`.stl`/`.glb`/`.gltf`), and `resolveAssetUrl(path)` anchors `package://`/relative refs onto the hub asset server (`http://<host>:8080/assets`). RobotModel keeps its own URL resolver for the UR `package://example-robot-data/...ur_description/` anchoring quirk; the generic `resolveAssetUrl` just strips the scheme and prepends the asset base.
- `core/uuid.ts` — context-safe UUID v4 used to mint tab/plot ids (`tabs.store`, `PlotTab`, `ThreeDTab`). Falls back from `crypto.randomUUID()` (secure-context only) to `getRandomValues` to `Math.random`, so it works when the app is served over plain HTTP from a LAN/VM address.
- `core/ik.ts` — pure damped-least-squares (Levenberg–Marquardt) IK math for RobotModel's `ik` joint source: `orientationError` (quaternion → axis·angle rotation vector), `solveDampedLeastSquares(J 6×N, e 6, λ)` → joint delta `dq = Jᵀ(J Jᵀ + λ²I)⁻¹e` via a small `solveLinear` (Gauss–Jordan). Only three.js vectors + number arrays, so it's node-testable and framework-light.
- `core/poseGizmo.ts` — reusable combined move+rotate gizmo (two `TransformControls`, rotate built first for pick priority, plane/outer-ring pickers pruned, `local` space), extracted from the `CoordinateFramePlugin` pattern. Used by `RobotIkController`; `CoordinateFramePlugin` still has its own inline copy (could adopt this later).

### `sdks/python` — minimal client
The SDK package is `sdks/python/webviz/`; the runnable examples all live under `sdks/python/demos/`. Each demo that uses the SDK adds the package's parent dir to `sys.path` at import (a 1-line bootstrap) so it runs straight from the repo without an install.
- `webviz/client.py` — two WS endpoints (mirror the binary header from `binary.ts`; require `pip install 'websockets>=11'` for the `websockets.sync` API). `webviz.Client` is the **source** side (`role=source`): advertises channels and sends JSON/binary frames. `webviz.Consumer` is the **client** side (`role=client`): the receive counterpart — `subscribe(name, callback)` routes JSON `message` frames to per-channel-name callbacks, deferring the subscribe until the channel is advertised (mirrors the app's `HubClient`); a background daemon thread drains the socket and binary frames are ignored (the consume path is JSON-only). `Consumer` is what powers the ROS adapter's reverse bridge.
- `demo_source.py` — dependency-free demo; publishes transforms/markers/battery, one of each remaining JSON 3D schema (LaserScan, OccupancyGrid, Path, Pose), and a rotating `wv/Log` stream (for the Log tab) via the hub's HTTP `/api/inject` instead of WS.
- `map_sim_demo.py` — dependency-free Map-tab (§11.5) demo. A `MapSim` holds a random world as hidden ground truth (border walls + random rectangular obstacles, `--seed` for repeatability) and drives a dot robot on a collision-avoiding random walk (publishing `mobile_base_link→odom` `wv/Transform`). It **raycasts** a 360° `wv/LaserScan` against the truth and **builds the published `wv/OccupancyGrid` from those scans** (cells traversed→free, hit→occupied, rest→unknown) so the map fills in SLAM-style as the robot explores; the trajectory is a `wv/Path` (`trail`). Also HTTP `/api/inject`. Drive the Map tab with fixed frame `odom`, map=`map`, scan=`scan`, path=`trail`, robot=`mobile_base_link`. The body frame is `mobile_base_link` (not the generic `base_link`) so it can run **alongside** `robot_demo.py` without colliding in the shared TF tree — both share the `odom` root, so both still resolve/render under the global fixed frame.
- `robot_demo.py` / `pointcloud_demo.py` / `image_demo.py` — WS demos built on `webviz.Client` (so they need `websockets`; `client.py` uses the `websockets.sync` API, which requires **websockets ≥ 11**): `robot_demo.py` feeds a UR5 `wv/RobotModel` (URDF + meshes fetched online over CORS, no local assets) + `wv/JointState` + base TF; `pointcloud_demo.py` streams animated binary `wv/PointCloud` frames; `image_demo.py` streams an animated RGB8 `wv/Image` (`camera_front`) for the Image tab. Each `*_payload()` duplicates the matching `binary.ts` payload layout.
- `ik_solver_demo.py` — the **consume-side** demo (uses both `webviz.Consumer` *and* `Client`; needs `numpy` + `websockets`), and the reference implementation of RobotModel's **external** IK backend. Parses a URDF (default the bundled `demo_arm`) into a serial chain, subscribes to the gizmo's `tcp_target` (`wv/Pose`), solves damped-least-squares Jacobian IK (an `IkChain` mirroring `core/ik.ts` — FK + geometric Jacobian + DLS, warm-started), and publishes `ik/joint_states` (`wv/JointState`). Meant as a template: swap `IkChain.solve` for MoveIt/KDL/ikfast. Assumes the target is in the robot base frame (set the app's fixed frame to the base link).

### `sdks/ros2` — ROS 2 adapter (§6.2)
A **drop-in** `ament_python` package (also runnable straight from the repo) that bridges ROS topics into the hub with zero robot-code changes, built on `sdks/python`'s `webviz.Client`. The node (`webviz_ros2_adapter/adapter.py`) **polls `get_topic_names_and_types()`** and auto-subscribes to every topic whose ROS type appears in the `REGISTRY` (`registry.py`), advertising one wv channel per topic (name = topic minus leading slash). `converters.py` holds the **pure** ROS→`wv/*` functions — it imports no `rclpy`/message packages (touches only message attributes), so it's node-testable with duck-typed stand-ins; binary converters (PointCloud2/Image/CompressedImage) return the post-header payload mirroring `binary.ts` (a **third** copy of the layout, alongside `client.py` and the demos). QoS is type-aware: sensor streams → best-effort `qos_profile_sensor_data`, `/tf_static` → transient-local, else reliable depth-10. `--include`/`--exclude` regexes filter topics. To support a new ROS type, add a `TypeEntry` + converter.

The optional **reverse bridge** (`--enable-reverse`, off by default since it lets the browser command the ROS graph) carries the small *fixed* set of **interactive** messages WebViz itself can emit back onto ROS topics. It keys on the wv channel *name* (not ROS type — the producer side is known and finite), driven by `registry.py`'s `REVERSE_REGISTRY` (a `ReverseEntry` per interactive channel): today the CoordinateFrame gizmo's `wv/Pose` (`tcp_target`) → `geometry_msgs/PoseStamped` on `/tcp_target`. The node opens a second `webviz.Consumer` (`role=client`) alongside its source `Client`, subscribes to each reverse channel, and on each frame builds + stamps + publishes the ROS message. Reverse converters live in `converters.py` as **`fill_*_from_wv(data, msg)`** functions that *mutate a passed-in ROS message* — so the module still imports no message packages and stays node-testable (the node owns the classes). To prevent a ROS→wv→ROS loop, reverse target topics are excluded from forward discovery. Override a topic with `--reverse-map CHANNEL=/topic`. Add a future interactive producer → add one `ReverseEntry` row.

### `sdks/cpp` — C++ SDK (§6.3)
A **header-only**, dependency-free source client (`include/webviz/client.hpp`) — POSIX/`ws://` only. It speaks just enough RFC 6455 over a raw TCP socket (own handshake + framing) to advertise channels and push frames, so it needs no Boost/OpenSSL/WS library; `#include` it and go. API mirrors `client.py`: `Client::advertise` → `Channel` with `send` (JSON, via a tiny `webviz::Value` builder — objects `{{"k",v}}`, arrays `webviz::arr({...})`), `send_binary`, `send_image`, `send_pointcloud`. **Zero-copy binary path**: the small frame headers are built on the stack and the bulk payload is emitted with one scatter-gather `writev()` — the camera/cloud buffer is never copied. To make that possible every client frame uses an **all-zero WebSocket mask key** (RFC 6455 needs the MASK bit, but a zero key is unmask-identity, avoiding a per-byte XOR copy; safe for a trusted LAN hub). A background thread drains inbound frames (answers ping, honors close). `test/test_payloads.cpp` asserts the encoded bytes match `binary.ts` (the cross-language contract; no network) and is a fourth copy of the payload layout. Build via CMake (`cmake -S . -B build && cmake --build build`; `ctest` runs the test); `examples/{transform,pointcloud}_demo.cpp` need a running hub.

## When changing the protocol

Edit `packages/protocol/src`, rebuild it, then update consumers. The Python SDK's binary header packing (`struct.pack("<B3xIdI", …)`) duplicates `binary.ts` — keep both in sync when the frame layout changes; likewise `pointcloud_demo.py`'s `pointcloud_payload()`, the ROS 2 adapter's `converters.py` (`pointcloud2_to_payload`/`image_to_payload`), and the C++ SDK's `client.hpp` (`append_binary_header`/`image_payload_prefix`/`pointcloud_payload_prefix`, guarded by `sdks/cpp/test/test_payloads.cpp`) duplicate the `wv/PointCloud` / `wv/Image` payload layouts. `PROTOCOL_VERSION` lives in `protocol/src/index.ts` and is sent in the `server_info` handshake.
